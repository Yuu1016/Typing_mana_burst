========================================================================
第2部：データベース設計（Entity設計）およびバックエンドロジック
========================================================================

1. テーブルおよびカラム設計
------------------------------------------------------------------------
データベースにはPostgreSQLを採用。マスタデータ（静的）とトランザクションデータ（動的）を明確に切り離して管理する。

■ マスター系テーブル（静的データ）
① `m_skills`（スキル・カードマスタ）
全スキルカードの基本パラメータを定義。
   - `skiil_id` (BIGINT, PK) : スキルID
   - `skill_name` (VARCHAR, UNIQUE) : スキル名（例：「ファイアバースト」）
   - `cost` (INT) : 消費コスト（1〜5）
   - `required_mana_type` (VARCHAR) : チャージに必要なマナ属性（`FIRE`, `WATER`, `LIGHT` など）
   - `base_value` (INT) : 基本威力（攻撃力や回復量のベース値）
   - `effect_type` (VARCHAR) : 効果の分類（`DAMAGE`, `HEAL`, `BUFF`, `DEBUFF`）
   - `target_type` (VARCHAR) : 対象（`ENEMY_SINGLE`, `ENEMY_ALL`, `PLAYER`）
   - `description` (TEXT) : スキルの説明文

② `m_stages`（ステージマスタ）
各ステージの難易度や出現エネミーの情報を定義。
   - `id` (INT, PK) : ステージID
   - `stage_number` (INT) : ステージ番号（1, 2, 3...）
   - `stage_name` (VARCHAR) : ステージ名
   - `enemy_name` (VARCHAR) : 敵の名前
   - `enemy_hp` (INT) : 敵のHP
   - `enemy_attack` (INT) : 敵の攻撃力
   - `limit_cost_pool` (VARCHAR) : 毎ターン発生するリミットコストの範囲（例: `3-5` などの文字列、プログラム側でパース）

■ トランザクション系テーブル（ユーザーデータ）
③ `t_users`（ユーザーテーブル）
プレイヤーの現在の進行度、永続ステータスを保存。
   - `id` (BIGINT, PK) : ユーザーID
   - `username` (VARCHAR) : ユーザー名
   - `current_hp` (INT) : 現在の最大HP（工房での強化反映後の値）
   - `gold` (INT) : 所持ゴールド（強化用のリソース）
   - `cleared_stage_id` (INT) : 現在クリア済みの最高ステージID

④ `t_user_decks`（ユーザーデッキテーブル）
ユーザーが現在編成しているスキルカードの紐付けを管理。
   - `id` (BIGINT, PK) : デッキ内一連番号
   - `user_id` (BIGINT, FK) : ユーザーID（`t_users.id` への参照）
   - `skill_id` (BIGINT, FK) : スキルID（`m_skills.id` への参照）
   - `slot_number` (INT) : デッキ内のスロット番号（1〜5）
   ※複合ユニーク制約：`user_id` + `slot_number`（同一スロットへの重複登録を防止）

⑤ `t_battle_logs`（バトルログテーブル）
バトルの結果、およびタイピング精度のログ保存用。
   - `id` (BIGINT, PK) : ログID
   - `user_id` (BIGINT, FK) : ユーザーID
   - `stage_id` (INT, FK) : 挑戦したステージID
   - `is_victory` (BOOLEAN) : 勝敗フラグ（TRUE = 勝利）
   - `clear_turns` (INT) : クリアに要したターン数
   - `total_typed_chars` (INT) : 総タイピング入力文字数
   - `missed_chars` (INT) : タイプミス数
   - `created_at` (TIMESTAMP) : プレイ日時

2. テーブル間のリレーションシップ（関係性）
------------------------------------------------------------------------
JPA（Hibernate）におけるEntity間の関係性定義の基準。

   [m_skills] 1 ─── N [t_user_decks] N ─── 1 [t_users]
                                                 │ 1
                                                 └─── N [t_battle_logs]
   [m_stages] 1 ────────────────────────────────┘ N

- `t_users` と `t_user_decks` （1対多 / `@OneToMany`）
  1人のユーザーは複数のデッキスロット（1〜5）を所有する。
- `m_skills` と `t_user_decks` （1対多 / `@ManyToOne`）
  各デッキスロットには、マスタに登録されたスキルが1つ割り当てられる。
- `t_users` と `t_battle_logs` （1対多 / `@OneToMany`）
  1人のユーザーは、過去のプレイログを複数蓄積する。

3. スキル効果・ロジック保持のためのエンジニア視点アドバイス
------------------------------------------------------------------------
特殊効果（持続ダメージ、条件付き威力増加など）の拡張性と、1〜2ヶ月の短期開発における実装スピードを両立するための設計アプローチ。

■ 提案A：【推奨】Enum（列挙型）× ストラテジーパターンの組み合わせ（シンプル＆堅牢）
DBの `m_skills.effect_type` にはシンプルな固定文字列（`DAMAGE`, `HEAL`, `POISON`, `LIMIT_BREAK_BOOST`）のみを格納。
Java側（Spring Boot）でこれを `Enum` として扱い、計算サービス等で分岐処理を行う。
