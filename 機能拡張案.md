# 【機能拡張アイデア】タイピング・マナ・バースト

基本開発完了後に検討すべき、追加機能およびアーキテクチャ改善のアイデアをまとめたものである。

---

## 1. ゲーム性を深める機能拡張（ドメインロジック）

### 敵のAIと反撃システム
- **概要**: 敵が単なる「的」ではなく、固有の行動パターンを持つようにする。
- **実装アイデア**:
  - `BattleService` 内に「敵の行動決定ロジック」を追加。
  - 特定の属性への耐性や、確率でプレイヤーにデバフ（例：一定時間タイピングの表示が隠れる「暗闇状態」など）を付与する仕組みを導入し、戦略性を高める。

### タイピング精度に基づくコンボ・倍率システム
- **概要**: プレイヤースキル（タイピングの正確さ・速度）をダメージに直結させる。
- **実装アイデア**:
  - リザルトで受け取っている `missedChars`（ミス数）やクリアタイムを評価。
  - ミスが少ない（またはノーミス）の場合、次ターンのダメージ倍率にボーナス（例：1.2倍〜1.5倍）をかけるロジックを追加。
  - 単なる視覚的エフェクトに留まらず、数値としての「コンボ」が繋がることで爽快感を向上させる。

### ガチャ（スキル召喚）システム
- **概要**: 貯めたゴールドの新たな使い道として、ランダムなスキルの獲得機能を追加する。
- **実装アイデア**:
  - 乱数（`Random`）と確率テーブル（マスタデータ）を組み合わせたガチャロジックの実装。
  - 未所持スキルを獲得した際に `m_skills` からプレイヤーの所持リストに追加する処理。

## 2. アプリの堅牢性を高める（アーキテクチャ・セキュリティ）

### 認証・認可の導入（Spring Security & JWT）
- **概要**: APIのセキュリティをプロ仕様に引き上げる。
- **実装アイデア**:
  - Spring Security と JWT（JSON Web Token）を導入。
  - 現在の「`userId` をそのまま送るだけで操作可能」な状態から、「正しくログインした本人のトークンによるリクエストか？」を裏側で検証する仕組みに変更する。

### グローバル例外処理（`@RestControllerAdvice`）
- **概要**: エラー発生時のフロントエンドへのレスポンスを綺麗に整形する。
- **実装アイデア**:
  - 現状の `throw new IllegalArgumentException(...)` などの例外をキャッチする専用クラスを作成。
  - 例外が発生した際、Spring Boot デフォルトの味気ないエラー画面ではなく、`{ "status": 400, "message": "コストが足りません" }` のような統一されたJSONフォーマットでフロントエンドに返すようにし、React側でのエラー表示制御を容易にする。

## 3. 開発フローの自動化（DevOps）

### GitHub Actions を使った CI/CD パイプライン
- **概要**: コードの品質を自動で担保する仕組みを作る。
- **実装アイデア**:
  - GitHub Actions を設定し、コードをコミットしてプッシュ（またはPull Request作成）するたびに、クラウド上で自動的に JUnit テストが走るようにする。
  - マージ前に「既存の機能が壊れていないか（テストが通っているか）」を自動チェックできる体制を構築し、エンバグを未然に防ぐ。

### フロントエンド
- **概要**:　フロントエンド修正点のメモ
- **実装アイデア**:
 - タイポ10回で０ダメージなどの制限を設ける
 - 最初のSELECT MAGICは打たなくていいっていう、打つべき文字と打たなくていい文字の区別
 - 空白は打たなくていい
 - タイピングする文字列はランダムな英語のword
 - Costの大きさによって、文字列の複雑さが増加する
 - Mana、Miss数とタイピングエリアが被ってるから場所を変える
 - 魔法を打ったら敵のターンに遷移
 - 魔法の同時選択によって、コストがちょうどになった時のコストボーナス（ダメージ増加）
 - 倒したらクリアになって、経験値が入る
 - レベルに合わせて、選択できる魔法が増えていく
