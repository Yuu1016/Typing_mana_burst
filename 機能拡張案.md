# 【機能拡張アイデア】タイピング・マナ・バースト

現在のMVP（最小限の遊べるプロダクト）完成後に追加・改善していくべきアイデアとロードマップをまとめたものである。

---

## 1. ゲーム性を深める機能拡張（フロントエンド・ドメインロジック）

### 魔法の同時選択と「JUSTコストボーナス」
*   **概要:** 1ターンに複数の魔法を組み合わせて発動できるようにする。
*   **実装案:** 選択したカードの合計コストが、現在の「上限マナ」とピッタリ一致した際、ダメージに倍率がかかる「JUSTボーナス（リミットブレイク）」を実装する。

### レベルアップによる恩恵（タイピングボーナス強化）
*   **概要:** プレイヤーの成長を実感できるパッシブスキルの実装。
*   **実装案:** 現在固定である「防衛スコア1につき3ダメージカット」などの数値を、レベルアップや工房での強化によって「5ダメージカット」などに引き上げ、強敵の攻撃を耐えやすくする。
*   レベルに合わせて、持ち込める（選択できる）魔法の種類やデッキ上限枠が増えていくシステムを追加。

### 敵のAIと固有ギミックの追加
*   **概要:** 敵が単なる「的」ではなく、固有の行動パターンを持つようにする[cite: 5]。
*   **実装案:** `BattleService` 内に「敵の行動決定ロジック」を追加[cite: 5]。特定の属性への耐性や、確率でプレイヤーにデバフ（例：一定時間タイピングの表示が隠れる「暗闇状態」など）を付与する仕組みを導入し、戦略性を高める[cite: 5]。

---

## 2. アプリの堅牢性を高める（バックエンド・アーキテクチャ）

### バトル計算ロジックのバックエンド完全移行
*   **概要:** 現在、ゲームのテンポを優先してフロントエンド（`Battle.tsx`）で行っている「防衛ダメージカット」などの計算を、本来の正しい設計であるバックエンド側に移管する。
*   **実装案:** APIエンドポイントを `/battles/attack` と `/battles/defense` のように分割し、フロントエンドでのタイピング結果（スコア）をバックエンドに送信して、サーバー側で最終的なダメージ計算とマナ計算を行って返すようにリファクタリングする。

### フロントエンドのコンポーネント分割（リファクタリング）
*   **概要:** 肥大化している `Battle.tsx` を整理する。
*   **実装案:** UIを `BattleVisualArea`, `BattleTypingArea`, `BattleHandArea` に分割し、複雑なタイマーやフェーズ管理ロジックは `useBattleManager` などのカスタムフックに切り出す。

### グローバル例外処理（`@RestControllerAdvice`）
*   **概要:** エラー発生時のフロントエンドへのレスポンスを綺麗に整形する[cite: 5]。
*   **実装案:** 現状の `throw new IllegalArgumentException(...)` などの例外をキャッチし、統一されたJSONフォーマットでフロントエンドに返すようにし、React側でのエラー表示制御を容易にする[cite: 5]。

---

## 3. 今後の大規模拡張（やり込み要素・DevOps）

*   **ガチャ（スキル召喚）システム:** 貯めたゴールドを使用し、乱数と確率テーブルを用いたガチャロジックで新しいスキルを獲得する仕組みを追加[cite: 5]。
*   **認証・認可の導入:** Spring Security と JWT を導入し、APIのセキュリティをプロ仕様に引き上げる[cite: 5]。
*   **CI/CDパイプライン構築:** GitHub Actions を設定し、コードのコミット時に自動でJUnitテストが走る体制を構築する[cite: 5]。
