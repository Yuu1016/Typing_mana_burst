
========================================================================
【バックエンド開発ロードマップ】タイピング・マナ・バースト
========================================================================
本作のデータベース定義（init.sql）および設計要件をベースとした、
Spring Boot（Java）によるバックエンド実装のステップ・バイ・ステップ手順。
GitHub や Sourcetree で細かく安全にコミット（粒度管理）できる構成。

------------------------------------------------------------------------
■ 基本開発方針
------------------------------------------------------------------------
1. マスタデータの参照（Read）から着手する
   - DB接続やJPAマッピングといった基礎部分の疎通を最優先。
   - 複雑なロジックを組む前に、インフラ・O/Rマッパーのバグを排除するため。
2. コアロジック（スキル効果計算）の独立化
   - DBアクセスやWeb(API)レイヤーから切り離し、純粋なドメインロジックとして実装。
   - JUnitを用いたユニットテストでロジックの正当性を保証する。

========================================================================
■ 詳細開発ステップ（全7段階）
========================================================================

Step 1: プロジェクト設定とDB接続の開通
------------------------------------------------------------------------
【概要】
アプリケーションが Docker 環境上の PostgreSQL と正常に疎通できる土台を構築する。

【対象・作成ファイル】
・src/main/resources/application.yml (または application.properties)
  - 接続先 URL、ユーザー名、パスワードの設定
  - JPA（Hibernate）の SQL ログ出力設定（`show-sql: true` 等）

【クリア基準】
・Spring Boot アプリケーションがエラーなく正常に起動すること。

【💡 コミット目安】
`chore: DB接続設定の追加`


Step 2: マスター系 Entity & Repository の実装
------------------------------------------------------------------------
【概要】
ゲーム中に変更されない静的なマスターデータ（スキル、ステージ）の読み込み基盤を構築する。

【作成クラス一覧】
・`Skill` (Entityクラス) : `m_skills` テーブルとマッピング
・`Stage` (Entityクラス) : `m_stages` テーブルとマッピング
・`SkillRepository` (Interface) : `JpaRepository<Skill, Long>` を継承
・`StageRepository` (Interface) : `JpaRepository<Stage, Long>` を継承

【クリア基準】
・テストコードや CommandLineRunner 等を利用し、`init.sql` で投入した全25件のスキル、
  全13件のステージデータが文字化けや型エラーなく取得できること。

【💡 コミット目安】
`feat: マスターデータ（Skill, Stage）のEntityとRepositoryを追加`


Step 3: トランザクション系 Entity & Repository の実装
------------------------------------------------------------------------
【概要】
ユーザー、デッキ、バトルログなど、動的に変動・蓄積するデータを扱う。
JPAにおけるテーブル間のリレーションシップ（@OneToMany, @ManyToOne 等）をここで定義。

【作成クラス一覧】
・`User` (Entity) : `t_users` テーブルとマッピング
・`UserDeck` (Entity) : `t_user_decks` テーブルとマッピング
・`BattleLog` (Entity) : `t_battle_logs` テーブルとマッピング
・`UserRepository` (Interface)
・`UserDeckRepository` (Interface)
・`BattleLogRepository` (Interface)

【リレーション定義の注意点】
・[m_skills] 1 ── N [t_user_decks] N ── 1 [t_users]
・[t_users] 1 ── N [t_battle_logs]
・[m_stages] 1 ── N [t_battle_logs]
※ `t_user_decks` の複合ユニーク制約（user_id + slot_number）をアノテーション等で考慮。

【クリア基準】
・初期データのユーザー（例: `typing_hero`）を取得した際、紐づく5枚のデッキスロット（UserDeck）情報が正しくJoin（またはLazy Load）されて取得できること。

【💡 コミット目安】
`feat: トランザクションデータ（User, UserDeck, BattleLog）のEntityとリレーションを追加`


Step 4: 【コアロジック基盤】 Enum と ストラテジーパターンの実装
------------------------------------------------------------------------
【概要】
本作最重要の設計要素。`m_skills.effect_type`（固定文字列）を Java 側で `Enum` として扱い、
拡張性の高いスキル効果の計算処理をストラテジーパターンで実装する。

【作成クラス一覧】
・`EffectType` (Enum)
  - `DAMAGE`, `HEAL`, `BUFF`, `DEBUFF` を定義
・`SkillEffectStrategy` (Interface)
  - `void apply(Skill skill, BattleContext context)` のような共通メソッドを定義
・`DamageStrategy` (Class) : 単体/全体へのダメージ計算、属性計算ロジック
・`HealStrategy` (Class) : プレイヤーのHP回復処理ロジック
・`SkillEffectFactory` (Class)
  - `EffectType` を引数に取り、対応する Strategy 実装クラスのインスタンスを返却する

【クリア基準】
・JUnit 等のユニットテストを記述し、ファクトリが正しいストラテジーを返却すること。
・ダメージ、回復の計算処理が仕様通り（基本威力や対象タイプに応じて）シミュレートできること。

【💡 コミット目安】
`feat: スキル効果計算用のEnumとストラテジーパターンを実装`


Step 5: Service レイヤーの実装（マスタ・ユーザー管理）
------------------------------------------------------------------------
【概要】
データの取得、加工、更新に関するビジネスロジックをカプセル化する。

【作成クラス一覧】
・`MasterDataService`
  - クライアント（フロントエンド）起動時・初期化時に必要なスキル一覧・ステージ一覧を一括取得して返却するロジック。
・`UserService`
  - ユーザープロフィールの取得
  - 工房でのゴールド消費による最大HP等の強化処理
  - デッキ編成処理（`t_user_decks` の対象スロットの `skill_id` を書き換えるバリデーション含むロジック）

【💡 コミット目安】
`feat: マスタデータ取得およびユーザー管理（デッキ編成・強化）のServiceを追加`


Step 6: Service レイヤーの実装（バトル進行）
------------------------------------------------------------------------
【概要】
Step 4 で構築したストラテジーパターンを活用し、ゲームのメインであるバトル進行を制御する。

【作成クラス一覧】
・`BattleService`
  - ステージ挑戦時の初期化（敵情報・リミットコストプールの決定）
  - ターン進行管理
  - スキル発動処理（`SkillEffectFactory` を呼び出し効果を反映。コスト計算含む）
  - バトル決着・リザルト処理（勝敗判定、獲得ゴールド計算、`t_battle_logs` への保存、ユーザーのクリアステージ更新）

【クリア基準】
・一連のモックデータまたは統合テストにおいて、スキル発動によってHPが変動し、最終的にバトル結果がログテーブルに永続化されること。

【💡 コミット目安】
`feat: バトル進行・スキル発動およびリザルト処理のServiceを追加`


Step 7: Controller（APIエンドポイント）の開放
------------------------------------------------------------------------
【概要】
フロントエンド（React / TypeScript）と通信するための REST API 窓口を設置する。

【作成クラス一覧】
・`MasterDataController` : `GET /api/v1/master-data`
・`UserController`       : `GET /api/v1/users/{id}`, `PUT /api/v1/users/{id}/decks`
・`BattleController`     : `POST /api/v1/battles/start`, `POST /api/v1/battles/cast`
・各種 DTO (Data Transfer Object)
  - 例: `UserResponseDto`, `SkillDto`, `BattleActionRequestDto` 
  ※Entity クラスをそのまま JSON として返却せず、必要な項目のみを内包した DTO に詰め替える（ベストプラクティス）。

【クリア基準】
・Postman、cURL、または VS Code の REST Client 等でエンドポイントを叩き、期待通りの JSON レスポンスがステータスコード 200（または 201）で返却されること。

【💡 コミット目安】
`feat: フロントエンド連携用の各種REST API ControllerとDTOを追加`
